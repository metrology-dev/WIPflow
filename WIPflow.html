Getting Started

WIP Flow runs entirely in your browser — no install, no server, no internet required. Open WIPflow.html from any folder and your data loads automatically. The app ships with 15 demonstration tasks; clear them under Settings → Data Management → Clear All Data when you're ready to start fresh.

Data is saved automatically on every change (400 ms debounce) and periodically in the background. A green dot in the sidebar confirms the last successful save. Use ↓ Save as HTML to produce a self-contained portable copy — see Saving & Portability below.

End Date Calculation

The calculated end date accounts for allocation and skips weekends and holidays.

calendar_days = ⌈ workdays ÷ (allocation% ÷ 100) ⌉

Starting from the start date, the app counts forward that many calendar workdays (Mon–Fri), skipping any dates in the holiday calendar.

Example: 10 workdays at 50% allocation needs 20 calendar workdays. Starting on a Monday with no holidays, the end date lands on the Friday four weeks later.

The end date field updates live as you type. Changing holidays in Settings immediately recalculates all existing task end dates.

Views

Dashboard — KPI summary cards, status/priority/lab bar charts, staff workload, and upcoming deadlines (next 14 days). Charts resize with the window.

Table — Sortable, filterable grid. Click any column header to sort; click again to reverse. Use the filter dropdowns and the search box to narrow results. Click any row to edit.

Gantt — Canvas timeline with three zoom levels: Day, Week, and Month. Hover over a bar for a full tooltip. Click Today to scroll to the current date. Bar colour = priority; fill opacity = progress.

Kanban — Cards grouped by status column. Drag a card to a different column to change its status. Filter by lab or person using the dropdowns above the board.

Saving & Portability

WIP Flow uses three complementary layers so your data is never lost and always travels with the file.

Layer 1 — localStorage (automatic)
Every change is saved to browser localStorage within 400 ms. The save indicator in the sidebar turns green after each successful write. This is fast and invisible. Limitation: localStorage is scoped to the file's URL. If you move WIPflow.html to a different folder, the browser opens a different origin and finds no data.

Layer 2 — Embedded data in the HTML file
On every save, the full dataset is written into a <script id="labwip-embedded-data"> tag inside the HTML file in memory. This means the file is self-contained — if you use ↓ Save as HTML, the downloaded copy carries all your data. Open it on any machine, any browser, without needing localStorage.

Layer 3 — .labwip JSON export
↓ Export .labwip (Settings → Data Management) downloads a plain JSON backup file. Use this to transfer data between files or as an off-site backup. Import with ↓ Import in the topbar.

Recommended workflow for portability:

  1. Work normally — localStorage handles day-to-day saves automatically.
  2. Periodically click ↓ Save as HTML in the sidebar. This downloads a versioned WIPflow_v1.4.N.html file with all data embedded.
  3. Replace your working WIPflow.html with the downloaded file.
  4. When moving to a new machine, copy that file — it carries everything.

Versioning

WIP Flow uses a three-part version number: MAJOR.MINOR.SAVE

PartWho changes itWhen
MAJORDeveloperBreaking changes or major new features
MINORDeveloperBug fixes, small improvements, new non-breaking features
SAVEApp (automatic)Increments each time you click ↓ Save as HTML

The current version is shown in the sidebar below the logo. When you download a copy via ↓ Save as HTML, the save counter increments and the filename includes the full version, e.g. WIPflow_v1.5.3.html. This means every exported file has a unique, monotonically increasing identifier — you can always tell which copy is newer just from the filename.

The save counter is stored in your data (not hardcoded), so it persists across sessions and survives file moves as long as you always start from the latest exported copy.

Tip: If you want to reset the save counter (e.g. after a fresh install), set saveVersion to 0 in the .labwip JSON file before importing.

Holiday Calendar

Holidays are excluded from workday counting. Add them individually in Settings → Holiday Calendar (enter a date and optional name, then click Add or press Enter), or import a CSV file.

CSV format: one row per holiday, first column is the date (YYYY-MM-DD), optional second column is the name. Comma or semicolon delimited. A header row is skipped automatically if the first cell is not a valid date.

2026-01-01,New Year's Day
2026-06-06,National Day
2026-12-25,Christmas Day

Adding or importing holidays immediately recalculates all task end dates.

Settings

All dropdown options used in the task editor (laboratories, persons, priorities, statuses, tags) are managed in Settings. Add items by typing and pressing Enter or clicking Add. Remove with ✕.

Removing an option does not modify existing tasks — tasks retain their values; the removed item just no longer appears in the dropdown for new tasks.

Autosave interval: set 1–60 minutes. Set to 0 to disable periodic saves (save-on-change still works).

Theme: Dark (default) or Light. Persisted across sessions.

Data Management:

  • Export CSV — spreadsheet-friendly view, export only (no import path).
  • Export Excel (.xls) — SpreadsheetML file, opens in Excel and LibreOffice Calc.
  • Print / Save as PDF — opens the browser print dialog; choose "Save as PDF" in the destination picker.
  • Export .labwip — full backup/transfer format; use Import to restore.
  • Clear All Data — irreversible; clears localStorage and resets all tasks and settings.

WIP Flow
Laboratory Work-in-Progress Tracker  ·  v1.6.0

WIP Flow is a standalone, offline-first web application for tracking work in progress in research laboratory environments. It runs entirely in the browser by opening a single HTML file — no installation, no server, no internet connection required after initial load.

Designed for operational planning in multi-laboratory settings: tracking ongoing work, visualising timelines, coordinating staff allocation, and estimating realistic end dates based on workday calculations including weekends and holidays.

Technical Architecture

Single HTML file (~3 500 lines). All HTML, CSS, and JavaScript inline. No build step, no package manager, no transpilation.

JavaScript is structured as plain object literals (no classes):

ModuleResponsibility
AppStateCentral data store; task CRUD; filtering/sorting
StoragelocalStorage, embedded data, file export/import
WorkCalendarWorkday arithmetic; end date calculation
AppView routing; autosave timer; lifecycle
TaskModalCreate/edit task dialog
DashboardKPI cards; canvas charts; ResizeObserver
TableViewSortable/filterable task grid
GanttCanvas timeline; scroll sync; tooltip
KanbanViewDrag-and-drop status columns
SettingsList editors; theme; autosave; holidays
ToastNon-blocking notifications

Canvas is used for Dashboard charts and the Gantt timeline. All other views are rendered via template-literal HTML strings assigned to innerHTML.
